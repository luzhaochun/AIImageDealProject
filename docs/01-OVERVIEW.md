# ImageDeal — 系统架构总览

> SaaS 图片协作平台：从 CSV 导入、批量入库、用户领图作图、一审/二审，到终稿归档的全流程系统。

---

## 1. 业务背景

平台解决 **大批量外链图片素材** 在团队内的协作生产问题：

1. 管理员批量导入 CSV（含图片 URL、分类）
2. 系统自动下载、校验、生成缩略图并分类入库
3. 用户按类目领图、在线编辑、提交审核
4. 审核员一审（可选二审）质量控制
5. 通过后进入终稿库，支持导出打包

核心驱动：**图片状态机** + **逻辑图库（虚拟队列）** + **多角色 RBAC**。

---

## 2. 架构风格选型

### 推荐：**前后端分离 + 模块化单体（Modular Monolith）**

| 方案 | 适用阶段 | 说明 |
|------|----------|------|
| **模块化单体** ✅ MVP | 0→1 | 一个代码库、一个部署单元，按 domain 分包，开发效率最高 |
| 微服务 | 1→N | 当导入/Worker/编辑器负载独立扩容时再拆 |
| BFF 层 | 可选 | 前端复杂时可加 Node BFF，初期 REST 直连即可 |

**不选微服务作为起点的原因：**

- 13 步流程高度耦合于 `images.status` 状态机
- 团队规模小、需求迭代快
- 异步任务（下载/缩略图）用 **独立 Worker 进程** 即可水平扩展，无需拆微服务

### 演进路径

```
Phase 1 (MVP)     单体 API + Worker + SPA
Phase 2 (增强)    编辑器、二审抽样、导出、通知
Phase 3 (扩展)    拆 Import Service / Media Service；CDN；WebSocket 待办
```

## 4. 技术栈总表

| 层级 | 技术 | 版本建议 | 选型理由 |
|------|------|----------|----------|
| 后端语言 | Go | 1.22+ | 高并发、静态编译、适合 IO 密集下载任务 |
| HTTP 框架 | **Beego v2** | 2.3.x | MVC 全栈、企业后台快速交付 |
| ORM | **Beego ORM** + MySQL Driver | — | 与框架一体；迁移用 golang-migrate |
| 异步队列 | **Asynq + Redis** | — | 批量下载/导出；Beego task 仅做 cron |
| 主库 | MySQL | 8.0+ | 事务、JSON（图层）、复杂状态查询、团队熟悉度高 |
| 缓存/队列 | Redis + Asynq | 7+ | 异步任务、领图分布式锁 |
| 对象存储 | MinIO（开发）/ 阿里云 OSS（生产） | — | S3 兼容 API |
| 认证 | JWT (HS256/RS256) | — | SPA 无状态认证 |
| 前端框架 | React + TypeScript | 18+ | 组件化、编辑器生态 |
| 构建工具 | Vite | 5+ | 快速 HMR |
| UI 组件库 | Ant Design | 5+ | 后台表格/表单/流程组件齐全 |
| 数据请求 | TanStack Query | 5+ | 服务端状态缓存、轮询任务进度 |
| 客户端状态 | Zustand | — | 轻量，存 token/用户信息 |
| 图片编辑 | Konva + react-konva | 二期 | 图层、裁剪、文字 |
| 容器化 | Docker Compose（开发）/ K8s（生产） | — | 基础设施标准化 |

## 6. 逻辑图库（虚拟队列）

图库不是物理文件夹，而是 **status + 角色 + 过滤条件** 定义的视图：

| 图库名称 | 过滤条件 | 可见角色 |
|----------|----------|----------|
| 原图库（待分配） | `status = pending_assign` | User |
| 原图库（已分类） | `status = pending_assign` + `category` | User |
| 已领取 | `status = assigned` + `assigned_to = me` | User |
| 废料堆 | `status = discarded` | Admin |
| 用户任务区 | `assigned_to = me` AND status IN (`assigned`,`in_progress`,`rejected`) | User |
| 一审队列 | `status = pending_1st_review` | Reviewer |
| 一审通过库 | `status = 1st_review_passed` | Reviewer, Admin |
| 二审队列 | `status = pending_2nd_review` | Reviewer |
| 终稿库 | `status IN (2nd_review_passed, completed)` | Admin |

## 8. 非功能需求（NFR）

| 维度 | 目标 |
|------|------|
| 可用性 | API 99.9%；Worker 任务至少一次投递 |
| 并发领图 | 同一图片不可被两人领取（DB 行锁 / Redis 锁） |
| 安全 | 租户隔离、JWT 过期、上传文件类型白名单 |
| 性能 | 列表分页；缩略图 CDN；编辑器懒加载 |
| 可观测 | 结构化日志、Prometheus 指标、任务失败告警 |
| 数据保留 | 版本历史可配置保留策略；废料堆定期清理 |
